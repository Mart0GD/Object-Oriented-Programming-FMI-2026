## Фундамент

В това упражнение ще се занимаваме с една много удобна функционалност на езика C++, а именно предифинирането на оператори. Но преди това, нека разберем защо изобщо ни е нужно нещо подобно. Нека вземем за пример следната структура:

~~~.cpp
struct Vector2D{
    Vector2D(double x, double y) : x(x), y(y) {};
    double x, y;
};

Vector2D add(const Vector2D& l, const Vector2D& r) { return {l.x + r.x, l.y + r.y}; }
Vector2D sub(const Vector2D& l, const Vector2D& r) { return {l.x - r.x, l.y - r.y}; }
double   dot(const Vector2D& l, const Vector2D& r) { return l.x * r.x + l.y * r.y;  }
~~~

Тази структура имплементира стандартни операции за манипулация и всичко е напълно валидно. Обаче възниква въпросът - какво става ако имам по сложен израз от вида: `((v1 + v2) * v3)`, като приемаме че * означава скаларно произведение? Кодът ни със сегашната си структура ще изглежда така:

~~~.cpp
int main(){
    Vector2D v1(0,0), v2(1,1), v3(13, 15);
    double res = dot(v3, add(v1,v2));
}
~~~

Това започва да става по-трудно за четене и съответно предпоставя грешки. Затова е редно да се запитаме дали не може ли да имаме просто нормален аритметичен израз, използвайки тези вектори. Искаме да получим нещо подобно:

~~~.cpp
int main(){
    Vector2D v1(0,0), v2(1,1), v3(13, 15);
    double res = (v1 + v2) * v3;
}
~~~

Отговорът на този въпрос е да, можем! Предефинирането на оператори ни позволява да придадем на някакви наши **потребителски** типове специално значение чрез стандартните оператори в езика. Като се целим нашите имплементации да са възможно най-близко до очакваното поведение на тези типове в света, който моделираме...
  
Но нека засега да дадем една стъпка назад и  разгледаме абстрактно нещата от високо...

## Какво характеризира една операция?

**Брой аргументи**

Една операция може да има произволен брой аргументи теоритично, но в езика C++ ще срещнем:

  - унарни - **!**, **++** ...
  - бинарни - **<<**, **>>**, **всички аритметични операции** ...
  - тернарни - само **?:**

Ето в този израз имаме всеки вид оператор: `a ? a + (-b) : -c`

**Приоритет**

Това са набор от математически правила, които ни гарантират ред на извършване на операциите. Като имаме приоритет сме сигурни, че в израз `1 + 2 * 4`, първо ще се изчисли ``2 * 4 = 8`` и след това ще се събере с ``8 + 1 = 9``.

**Асоциативност**

Когато в рамките на даден израз участват операции с еднакъв приоритет, за разрешаване на последователността на изпълнение на операциите се използва тяхната асоциативност. Операциите могат да бъдат **ляво** или **дясно** асоциативни.
  
Ето пример:  
В израза ``5 – 3 + 2``, операциите събиране (+) и изваждане (–) имат еднакъв приоритет. Те са ляво асоциативни, което означава, че първо ще бъде извършено изваждането ``5 – 3 = 2``, след което ``2 + 2 = 4``.

**Позиция на оператора спрямо аргументите**

Една операция в нашия контекст може да бъде една от следните:

  - префиксна - **++a**, **-a** ...
  - постфиксна - **a++**, **a->** ...
  - инфиксна   - **a + b**, **a - b** ...


## Кои оператори може/не можем да предефинираме в C++

В тази табличка изчерпателно са показани всички оператори, които могат да се предефинират в езика **C++**

| Категория                    | Оператори                                      | Бележки |
|-----------------------------|------------------------------------------------|--------|
| Аритметични (унарни)        | +, -                                           | Унарни версии |
| Аритметични (инкремент)     | ++, --                                         | Префиксни и постфиксни |
| Аритметични (бинарни)       | +, -, *, /, %                                  | Стандартни |
| Сравнение                   | ==, !=, >, <, >=, <=                          | |
| Логически                   | &&, \|\|, !                                   | |
| Побитови                    | &, \|, ^, ~, >>, <<                           | |
| Указатели                   | *, ->, ->*, &                                 | Дереференция, достъп |
| Индексиране                 | []                                             | |
| Функционално извикване      | ()                                             | functor |
| Преобразуване на тип        | (type)                                         | cast оператор |
| Управление на памет         | new, delete                                    | |
| Запетая                     | ,                                              | |

**Какво не можем да предефинираме?**  
  
Не може да се предефинират операторите следните оператори:

  - .  (достъп до член данни/функции на обект)
  - .* (достъп до член данни на обект чрез указател към тях)
  - :: (scope оператора)
  - ?: (тернарният оператор)
  - sizeof, typeid

Първите три не можем да предефинираме от логически съображения, тъй като най-верятно потенциално предефиниция ще доведе до много сериозни грешки, тъй като имплементацията им не е елементарна отдолу. 

Тернарният оператор не може да предефинираме поради факта, че само един от аргументите му се изчислява и типът на връщане не е ясен по време на компилация

Последните две, както и всички други ключови думи и препроцесорни директниви не могат да се предефинират, тъй като работят по време на компилация, единствените запазени думи, които предефинираме са new и delete, тъй като те имат runtime действие...
