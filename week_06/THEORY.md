## Процедурно програмиране vs ООП

Досега кодът, който сме писали е  бил единствено и само процедурен, но какво значи това?

**Процедурното програмиране** е парадигма, това е начин на писане на код, при който данните и функционалността са отделни. Всяка функция, всеки блок от код си е самостоятелна единица, която извършва някаква логика/алгоритъм при подадени данни. Самата ни програма съдържа инструкции и групи от инструкции, наречени функции. Тук имаме различните понятия като рекурсия, логически конструкции if/else и други, които ръководят хода на нашата програма. 

**ООП** програмиране от друга страна е също така парадигма, която диктува как пишем нашия код, но тук да данните и функционалността са окомплектовани в някакви носители, които ще наричаме **класове**. Тези обекти знаят състоянието на техните данни и имат функционалността, с която да го променят, като важната функционалност за промяна е скрита от външния свят и външият потребител получава някакъв интерфейс, с който да променя или да получава достъп до данните в обекта. 

Клас като дума е нова за нас. В обектно ориентираното програмиране класът съдържа всички характеристика на някакъв обект, като обект е конкретна инстанция със собствени стойности на данните и имплементации на функциалността. Така че класовете като представители на ООП парадигмата, обединяват както поведението така и данните на елемнти от нашата програма. 

---

**Както може би сте чували в Обектно ориентираното порграмиране имаме 4 базови парадигми:**

**Енкапсулация**  
Това е най-отличителния и най-често нарушаван принцип на ООП. Всеки един друг принцип може да бъде реализиран и извън ООП базиран код, освен този. Идеята на енкапсулацията е нашите данни да зависят и да са известни само на затворената функционалност на нашия клас. Външният свят трябва да познава само функционалността, която ние експлицитно сме изнесли за работа с обекти от такъв клас. Това ни позволява да имаме контрол над валидността на един обект и да знаем какво е състоянието му във всеки един момент от работата на програмата, за да може въняният свят да не се интересува дали обектът работи или не...
  

**Наследяване/Преизползване**  
Един клас може да разшири друг вече съществуващ, без да знае каквито и да е детайли за функционалността или данните му. Така универсално може да надграждаме и добавяме нова поведение към нашите обекти...

  
**Полиморфизъм**  
Една и съща функционалност може да бъде преизползвана с различни по тип данни, както видяхме последния път с шаблоните (**статичен полиморфизъм**). Или функционалност на даден обект може да зависи от това как е имплементирана на ниво клас, това ще видим при реализирането на виртуални функции (**динамичен полиморфизъм**). Общо взето едни и същи действия с различни обекти могат да доведат до различен изход...
  
  
**Абстракция**  
Нашият клас трябва да бъде като черна кутия за вънишния свят. Трябва да предоставя функционалност за работа с данните окомплектовани вътре в даден обект, но не трябва външния потребител да мисли за логиката, която е имплементирана вътре. Нашите данни са разделени от функционалността, която ги използва...

## Как пишем класове в C++  
  
Тъй като в този курс пишем на C++, ще разгледаме какви са особеностите на работа с обектно ориентиран код в този език. 

**Деклариране и дефиниране на класове**

За да декларираме клас ще използваме запазената дума **class** последвано от име, което не е засължително (създава анонимен клас) и след това поставяме къдрави скобки -> {}
В един клас, както при структурите можем да имаме произволен както по брой така и по тип полета, тях ще наричаме член данни на класа. 

Нека разгледаме как може да си дефинираме клас кола:

~~~.cpp
class car // --> декларация на класа <class> + <name>
{  
    int year_of_production; // --> първо поле от тип int
    char brand[32];         // --> второ поле от тип char*
    color paint_job;        // --> трето поле от тип color (енумерация)
    // ...
}
~~~

Както и при структурите и тук можем да заложим стойности по подразбиране на член данните на нашия клас, които да бъдат задаени на нашия обект ако ние не сме подали такива.
  
Ето как:  

~~~.cpp
class car // --> декларация на класа <class> + <name>
{
    // може да се зададат и с къдрави скобки {value}

    int year_of_production = 1992; 
    char brand[32] = "Honda";             
    color paint_job = color::RED;        
    // ...
}
~~~

**Член функции**

Както се сещате досега обработвахме структури и извършвахме някакви операции с техните данни посредством външно дефинирани функции, както в ето този пример:
~~~.cpp
struct rational_number {
    int a; // цяла част на числото
    int b; // дробна част на числото
};

void add(rational_number& l, const rational_number& r)
{
    l.a = l.a * r.b + l.b * r.a;
    l.b = l.b * l.b;
}
void sub(rational_number& l, const rational_number& r)
{
    l.a = l.a * r.b - l.b * r.a;
    l.b = l.b * l.b;
}
~~~

Това е напълно валидно и спазва принципите на процедурното програмиране, данните ни са разделени от функционалността. Add знае, че му трябва две структури и че трябва да ги събере. Единственото, кеото може да не ви допадне е, че функциите имат директен достъп до данните на rational_number, което е в сблъсък с ООП принципа за енкапсулация...

Ако искаме да напишем същия този код, спазвайки ООП принципите ще трябва декларираме функциите като **член функции** на класа **rational_number**. Но какво значи това ще се запитате най-вероятно. Ако напишем една функция вътре в клас тя става челн функция за този клас, което означава, че има достъп до виски член данни на класа и може да извършва промени, проверки или заявки за информация към тях. Това дали решава напълно нашия **ООП** казус, нека видим? 

~~~.cpp
class rational_number{
    int a;
    int b;

    void add(rational_number& r){
        a = a * r.b + b * r.a;
        b = b * r.b;
    }

    void sub(rational_number& r){
        a = a * r.b - b * r.a;
        b = b * r.b;
    }
};

// дефиниране на два обекта 
rational_number num1 = {1,2};
rational_number num2 = {1,2};

num1.add(num2); // достъп до член функция
~~~

**This указателя**

В момента функциите и техните дефиниции лежат стриктно в класа rational_number, което ни улеснява със сигурност, но надявам се сте се запитали, защо имаме достъп до полетата на класа, това някаква магия ли е? Ако сте се запитали това на прав път сте само че няма никаква магия. При извикването на `num1.add(num2);` неявно се подава като параметър и указател към обекта **num1**, така че извикването в действителност е:

`num1.add(&num1, num2);`

**Неявно/Явно използване на this**
  
Въпреки че остановихме съществуването на втори неявен параметър на член функцията add, това все още не ни отговаря на въпроса защо може да използваме **a** и **b** без допълнителен контекс? Това се получава поради неявното използване на this, за достъп до челен данни и функции, в действителност кода изглежда така:

~~~.cpp
void add(rational_number* const this, rational_number& r){
    this->a = this->a * r.b + this->b * r.a;
    this->b = this->b * r.b;
}
~~~

Указателят към this е константен и не можем да го променяме. Този код, който разписах е явна употреба на this указателя. Това ще ви се налага ако имате сблъсък на имена във функцията или просто решите да използвате this навсякъде, никой не ви спир, единствено е лош стил...

**Пример за колизия на имена**
~~~.cpp
// този код ще извърши операциите върху подадените аргумнети a и b и следователно няма да промени нищо в обекта
void add(int a, int b){
    a = a * b + b * a;
    b = b * b;
}

// заради това използваме експлицитно this 
void add(int a, int b){
    this->a = this->a * b + this->b * a;
    this->b = this->b * b;
}
~~~
