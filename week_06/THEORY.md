## Процедурно програмиране vs ООП

Досега кодът, който сме писали е  бил единствено и само процедурен, но какво значи това?

**Процедурното програмиране** е парадигма, това е начин на писане на код, при който данните и функционалността са отделни. Всяка функция, всеки блок от код си е самостоятелна единица, която извършва някаква логика/алгоритъм при подадени данни. Самата ни програма съдържа инструкции и групи от инструкции, наречени функции. Тук имаме различните понятия като рекурсия, логически конструкции if/else и други, които ръководят хода на нашата програма. 

**ООП** програмиране от друга страна е също така парадигма, която диктува как пишем нашия код, но тук данните и функционалността са окомплектовани в някакви носители, които ще наричаме **класове**. Тези обекти знаят състоянието на техните данни и имат функционалността, с която да го променят, като важната функционалност за промяна е скрита от външния свят и външият потребител получава някакъв интерфейс, с който да променя или да получава достъп до данните в обекта. 

Клас като дума е нова за нас. В обектно ориентираното програмиране класът съдържа всички характеристика на някакъв обект, като обект е конкретна инстанция със собствени стойности на данните и имплементации на функциалността. Така че класовете като представители на ООП парадигмата, обединяват както поведението така и данните на елемнти от нашата програма. 

---

**Както може би сте чували в Обектно ориентираното порграмиране имаме 4 базови парадигми:**

**Енкапсулация**  
Това е най-отличителния и най-често нарушаван принцип на ООП. Всеки един друг принцип може да бъде реализиран и извън ООП базиран код, освен този. Идеята на енкапсулацията е нашите данни да зависят и да са известни само на затворената функционалност на нашия клас. Външният свят трябва да познава само функционалността, която ние експлицитно сме изнесли за работа с обекти от такъв клас. Това ни позволява да имаме контрол над валидността на един обект и да знаем какво е състоянието му във всеки един момент от работата на програмата, за да може въняният свят да не се интересува дали обектът работи или не...
  

**Наследяване/Преизползване**  
Един клас може да разшири друг вече съществуващ, без да знае каквито и да е детайли за функционалността или данните му. Така универсално може да надграждаме и добавяме нова поведение към нашите обекти...

  
**Полиморфизъм**  
Една и съща функционалност може да бъде преизползвана с различни по тип данни, както видяхме последния път с шаблоните (**статичен полиморфизъм**). Или функционалност на даден обект може да зависи от това как е имплементирана на ниво клас, това ще видим при реализирането на виртуални функции (**динамичен полиморфизъм**). Общо взето едни и същи действия с различни обекти могат да доведат до различен изход...
  
  
**Абстракция**  
Нашият клас трябва да бъде като черна кутия за вънишния свят. Трябва да предоставя функционалност за работа с данните окомплектовани вътре в даден обект, но не трябва външния потребител да мисли за логиката, която е имплементирана вътре. Нашите данни са разделени от функционалността, която ги използва...

## Как пишем класове в C++  
  
Тъй като в този курс пишем на C++, ще разгледаме какви са особеностите на работа с обектно ориентиран код в този език. 

**Деклариране и дефиниране на класове**

За да декларираме клас ще използваме запазената дума **class** последвано от име, което не е засължително (създава анонимен клас) и след това поставяме къдрави скобки -> {}
В един клас, както при структурите можем да имаме произволен както по брой така и по тип полета, тях ще наричаме член данни на класа. 

Нека разгледаме как може да си дефинираме клас кола:

~~~.cpp
class car // --> декларация на класа <class> + <name>
{  
    int year_of_production; // --> първо поле от тип int
    char brand[32];         // --> второ поле от тип char*
    color paint_job;        // --> трето поле от тип color (енумерация)
    // ...
}
~~~

Както и при структурите и тук можем да заложим стойности по подразбиране на член данните на нашия клас, които да бъдат задаени на нашия обект ако ние не сме подали такива.
  
Ето как:  

~~~.cpp
class car // --> декларация на класа <class> + <name>
{
    // може да се зададат и с къдрави скобки {value}

    int year_of_production = 1992; 
    char brand[32] = "Honda";             
    color paint_job = color::RED;        
    // ...
}
~~~

**Член функции**

Както се сещате досега обработвахме структури и извършвахме някакви операции с техните данни посредством външно дефинирани функции, както в ето този пример:
~~~.cpp
struct rational_number {
    int a; // цяла част на числото
    int b; // дробна част на числото
};

void add(rational_number& l, const rational_number& r)
{
    l.a = l.a * r.b + l.b * r.a;
    l.b = l.b * l.b;
}
void sub(rational_number& l, const rational_number& r)
{
    l.a = l.a * r.b - l.b * r.a;
    l.b = l.b * l.b;
}
~~~

Това е напълно валидно и спазва принципите на процедурното програмиране, данните ни са разделени от функционалността. Add знае, че му трябва две структури и че трябва да ги събере. Единственото, кеото може да не ви допадне е, че функциите имат директен достъп до данните на rational_number, което е в сблъсък с ООП принципа за енкапсулация...

Ако искаме да напишем същия този код, спазвайки ООП принципите ще трябва декларираме функциите като **член функции** на класа **rational_number**. Но какво значи това ще се запитате най-вероятно. Ако напишем една функция вътре в клас тя става челн функция за този клас, което означава, че има достъп до виски член данни на класа и може да извършва промени, проверки или заявки за информация към тях. Това дали решава напълно нашия **ООП** казус, нека видим? 

~~~.cpp
class rational_number{
    int a;
    int b;

    void add(rational_number& r){
        a = a * r.b + b * r.a;
        b = b * r.b;
    }

    void sub(rational_number& r){
        a = a * r.b - b * r.a;
        b = b * r.b;
    }
};

// дефиниране на два обекта 
rational_number num1 = {1,2};
rational_number num2 = {1,2};

num1.add(num2); // достъп до член функция
~~~

**Разделна компилация на член функции**

Както виждаме вече член данните са окомплектовани в класа както и самите функции, енкапсулация в случая е постигната. Това е един начин да дефинираме член функции обаче. Ако наш клас има 100 такива, че и по-дълги методи, то става невъзможно за четене ако ги напишем всичките в header файла. От тук нататък, ако някоя член функция е по-голяма от 2 реда ще пишем дефиницията и в отделен .cpp файл.


~~~.cpp
// .hpp
class rational_number{
    int a;
    int b;

    void add(rational_number& r);
    void sub(rational_number& r);
};

// ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

// .cpp
void rational_number::add(rational_number& r){
    a = a * r.b + b * r.a;
    b = b * r.b;
}

void rational_number::sub(rational_number& r){
    a = a * r.b - b * r.a;
    b = b * r.b;
}
~~~  

    
Ще използваме оператора **::** както при namespace-овете, за да укажем принадлежност на дефиницията към конкретна декларация. В случая функциите са дефиниции на декларираните в .hpp член функции add и sub на **rational_number**.

**This указателя**

В момента функциите и техните дефиниции лежат стриктно в класа rational_number, което ни улеснява със сигурност, но надявам се сте се запитали, защо имаме достъп до полетата на класа, това някаква магия ли е? Ако сте се, на прав път сте, обаче няма никаква магия.   
  
При извикването на num1.add(num2) неявно се подава като параметър и указател към обекта **num1**, така че извикването в действителност е:

`num1.add(&num1, num2);`

**Неявно/Явно използване на this**
  
Въпреки че остановихме съществуването на втори неявен параметър на член функцията add, това все още не ни отговаря на въпроса защо може да използваме **a** и **b** без допълнителен контекс? Това се получава поради неявното използване на this, за достъп до челен данни и функции, в действителност кода изглежда така:

~~~.cpp
void add(rational_number* const this, rational_number& r){
    this->a = this->a * r.b + this->b * r.a;
    this->b = this->b * r.b;
}
~~~

Указателят към this е константен и не можем да го променяме. Този код, който разписах е явна употреба на this указателя. Това ще ви се налага ако имате сблъсък на имена във функцията или просто решите да използвате this навсякъде, никой не ви спирa, но е проява на лош стил...

**Пример с колизия на имена**
~~~.cpp
// този код ще извърши операциите върху подадените аргумнети a и b и следователно няма да промени нищо в обекта
void add(int a, int b){
    a = a * b + b * a;
    b = b * b;
}

// заради това използваме експлицитно this 
void add(int a, int b){
    this->a = this->a * b + this->b * a;
    this->b = this->b * b;
}
~~~

**Оператори за достъп**

Чрез член функции, можем да реализираме достъп за промяна и четене на данните на конректен обект. Обаче трябва да запазим **инварианта** на обекта, като инвариант значи своитво, което винаги е вярно. В случая на rational_number, ако добавя нова член функция div(...), ще трябва да се съобрази, че числото което е подадено като делител не трябва да бъде **0!!**.   
  
Ето няколко начина как да реализираме тази функционалност:

~~~.cpp

// Първи вариант --> прекратяване на операцията. 
// Този подход не дава на външния свят никаква информация какво е станало с изпълнението
void rational_number::div(const rational_number& rhs)
{
    if(rhs.a == 0 && rhs.b == 0) return;

    this->a = this->a * rhs.b;
    this->b = this->b * rhs.a;
}

// Втори вариант --> връщане на стойност 1/0
// Този подход е по-добър от предния, тъй като външния свят може да обработи грешката.
bool rational_number::div(const rational_number& rhs)
{
    if(rhs.a == 0 && rhs.b == 0) return false;

    this->a = this->a * rhs.b;
    this->b = this->b * rhs.a;
    return true;
}


// Трети вариант --> промяна на невалидната стойност на валидна
// Този трябва да бъде описан за външния свят, за да се знае, за променящото своиство на функцията (rhs вече не е const)
void rational_number::div(rational_number& rhs)
{
    if(rhs.a == 0 && rhs.b == 0) rhs.a = 1, rhs.b = 1;

    this->a = this->a * rhs.b;
    this->b = this->b * rhs.a;
}
~~~

В общия случай ще пишем операции за достъп до елементи (гетъри) и операции за промяна на елементи (сетъри) само ако имаме нужда от тях, тъй като рано или късно ще нарушим енкапсулацията, ако направим всички член данни променими или достъпни, идеята на ООП е да скрием колкото се може повече ненужна информация от външния свят, силата идва в абстракцията:

Ето примери за това как можем да имплементираме get() и set() функционалност:

~~~.cpp
// .hpp
class student {

    bool set_name(const char* new_name);
    const char* get_name() const { return name; }

    bool set_age(int new_age);
    int get_age() const { return age; }

    bool set_weight(double new_weight);
    double get_weight_in_bag() const { return weight_in_bag; }

    char* name;
    int age;
    double weight_in_bag;
}

// ~~~~~~~~~~~~~~~~~~~~~~
// .cpp

bool student::set_name(const char* new_name)
{
    if(!new_name || *new_name == '\0') return false;

    char* temp = new(std::nothrow) char[strlen(new_name) + 1];
    if(!temp) return false;

    strcpy(temp, new_name);
    delete[] this->name;
    this->name = temp;

    return true;
}

bool student::set_age(int new_age)
{
    if(new_age < 7 || new_age > 99) return false;
    this->age = new_age;
}

bool student::set_weight(double new_weight)
{
    if(new_weight < 0 || new_weight > 999) return false;
    this->weight_in_bag = new_weight;
}
~~~

Това е добър пример за запазване на инвариант, но в най-добрия слуай, не искаме всички полета на класа да са променими и достъпни, защото това убива цялостната идея на енкапсулацията!
  
Когато поставим думата const след декларацията и дефиницията на член функция, тогава указваме на функцията, че тя ще запази инварианта на обекта, няма да проемни негови член данни!

**Създаване на обекти**

Досега създавахме обекти от тип някаква структура, използвайки инициализиращия списък на структурата или пишехме своя функция, която да създаде валидна структура по тези данни, като в ето този пример:

~~~.cpp
// .hpp
struct pot{
    char flower_name[32];
    int age;
};

// ......................
// .cpp

bool create_pot(pot& obj, const char* name, int age)
{
    if(strlen(name) > 31) return false;

    strcpy(obj.flower_name, name);
    obj.age = age;
}
~~~

**Конструктор**  
  
Това е валидно и като отделна функция и като член функция на структурата/класа pot. Затова в C++ съществува понятието **конструктор**. Това е член функция, без тип на връщнае, която носи същото име като това на класа/структурата, в която се помещава. Ролята на конструктура е да създаде валиден обект от типа на класа/структурата, който да е готов за обработка от външния свят. Най-важната подробност за конструкторите е, че ако не могат да създадат обект, те трябва да спрат работата си да зачистят потенциално заделена памет и да хвърлят някакво изключение, но ние ще се занимава с изключения малко по нататък, затова засега просто връщайте флаг, който ще инвалидира вашия обект.

Ето пример за използване на конструктор с досегашните ни знания:

~~~.cpp
//.hpp

class flower{
    // това тук е декларацията на конструктора на класа flower, той отговаря за създаването на валидни обекти от тип flower!
    flower(const char* name, int age, color pot_color);
    
    char* name;
    int age;
    color pot_color;

    bool valid;
};

// ....................
// .cpp

// това тук е дефиницията на конструктора
flower::flower(const char* name, int age, color pot_color)
    : valid(true) // --> това тук е инициализиращият списък на конструктора. Тук можем да зададем стойности на полета, които трябва да си знаят стойността compile time, като константите
{
    if (pot_color == color::UNKNOWN || !name){
        valid = false;
        return;
    }

    this->name = new(std::nothrow) char[strlen(name) + 1];
    if(!this->name){
        valid = false;
        return;
    }
    
    strcpy(this->name, name);
    this->age = age;
    this->pot_color = pot_color;
}
~~~

**Деструктор**

Деструкторър, досъщ като конструктора е член функция на класа/структурата, която няма тип на връщане и има специално предназначение. Както можете да се досетите деструкторът трябва да свърши обратната работа на конструктора, а именно да зачисти данните и състоянието на обекта.

Деструкторът има почти същата дефиниция като конструктора само че салгаме едно **~** отпред. В случая деструкторът на предишния клас ще бъде:

~~~.cpp
flower::~flower()
{
    delete[] name;
    name = nullptr;
}
~~~

**Struct vs Class**

финално нека разгледаме тези две дефиниции, едната на клас, другата на структура:

~~~.cpp
struct A
{
    int val;
};

class B
{
    int val;
}

A foo;
B bar;

foo.val = 1;
bar.val = 2;
~~~

На пръв поглед няма разлика между тях, следователно класовете и структурите са взаимо-заменяеми, но тогава защо ви го показвам този пример изобщо?  
  
Причината е, че тези два обекта не са съвсем едни и същи, по дефиниция структурите са изцяло public, а класовете са private. Какво означават тези думи ни предстой да разберем. Другата разлика между тях е принципна, от тук нататък ако искате да окомлектовате някакви данни без много сложна функционалност към тях използвайте структура, ако обаче ще изпълнявате сложна логика и манипулации на данните ви, използвайте класове...

**Видимост**

**Private**
  
Това е запазена дума в езика, ако я използваме в клас всички член данни и функции ще бъдат достъпни само и единствено от други член функции на същия клас, тази дума ни гарантира, че външния свят няма достъп до нашите данни и начини за промяна на обекта без ни да му го предоставим.

**Public**

Това е друга запазена дума в езика, която ни позволява да направим всички член данни/функции под нея достъпни за външния свят. Под тази дума стои интерфейсът за работа с обекти от класа ни.

~~~.cpp
class A{
private:
    // по подразбиране
};

struct B{
public:
    // по подразбиране
};

// Пример как би изглеждал един клас за динамичен масив от целочислени стойности...
class array {
    int* data;
    int  cnt, size;

public:

  array(int size);
  bool add(int elem);
  // ....
};
~~~
