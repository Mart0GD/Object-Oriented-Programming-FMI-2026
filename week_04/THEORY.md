## Леко обобщение до момента...

В предишното упражнение се занимавахме с файлове с последователен достъп. Това означаваше, че данните му се четат последователно от началото на файла 
до неговият край. Това реализирахме като към някакъв текстови файл асоциирахме поток от **fstream** в стандартен (**текстови**) режим.     
    
Можехме ли да **добавим** диркетно информация на дадена позиция - **уви не...**  
  
1) Можем да презапишем информацията от файла в нов, да извършим корекцията и да допишем останалото, като изтрием 
първоначалния файл. Скъпа операция, но работеща.  
2) Мърдаме всички записи с едно назад, както правехме при изтриването в масиви по УП, но постоянното преместване на указателя в текстови 
файл е скъпо, поради нуждата да заредим файла наново и да отместим указателя спрямо началото при всяко стъпка.

Можем ли да **редактираме** данни директно на дадена позиция - **отново не...**    

1) Ако данните за редакция са с еднакъв размер, презаписваме върху старото съдържание.
2) Ако данните ни не са толкова перфетни, можем единствено да презапишем наново файла, като извършим редакцията на желаното от нас място, тъй като при подход 1 ще 
трябва да пишем върху данни, които не искаме да редактираме 

Можем ли да **изтрием** данни от файл с последователен достъп - **както вече се досещате, не...**  
  
1) Изчитаме файла докато прескочим желаната позиция за изтриване и да мърдаме последователно съдържанито с едно назад.  
2) Създаваме нов файл, презаписваме съдържанието до желаната за изтриване позиция, пропускаме съдържанието, което не искаме и презаписваме останалото.  

Както виждате файловете с последователен достъп, не ни устройват особено много, когато искаме да извършваме операции за редактиране, изтриване или добавяне. Затова 
нека въведем една нова концепция.

## Файлове с директен достъп

Както знаем файловете с директен достъп ни позволяват да прескачаме на прочизволни места, без нуждата да се изчитат данните последователно. Записите ни са **произволни по размер** и **неформатирани**. Затова имаме нужда от някакви външни знания за формата.
C++ не ни гарантира такива под никаква форма, затова ние сами и ще си ги определяме. В пределите на курса ще се сблъскаме с различни подходи за формат, както например:

  - Файлът да е разделен на блокове с еднакъв размер.
  - Преди всеки запис да има метаданни, които оказват размера му.
  - Всеки запис във файла да е с еднакъв размер.
  - и още много...



## Работа с двоични файлове в C++  

В C++ ще работим с файловете в двоичен режим, което гарантира, че данните ще бъдат прочетени или записани без
каквито и да е преобразувания и интерпретации. Двоичният режим позволява преносимост на
файловете между различни операционни системи, тъй като няма преобразуване, данните могат да
бъдат възстановени коректно на различни платформи, съблюдаващи формата им.
   
### Отваряне 
  
Както при текстовите файлове и тук ще искаме да асоцираме даден файл към поток, за да получим буфер, от който да можем да четем и записваме. За работа с файлове
с директен достъп отново ще изполваме класовете за потоци дефинирани във ``fstream``, като едиснтвената разлика с текстовите файлове е, че тук ще добавяме разширение за четене в **двоичен режим**, тъй като всеки файл без него, по подразбиране е текстови.

~~~.cpp
// Отваряне на файл в двоичен режим

#include <fstream>

int main(void){

  // Отваряме поток за запис или четене в двоичен режим
  std::ofstream os("File.bin", std::ios::binary);
  if(!os.is_open()) return -1;
  
  return 0;
}
~~~

### Четене и писане 

При двоичните файлове четенето и писането с операторите `>>`, `<<` и функцията `getline()` **не е ефективно**, тъй като:

- не искаме **форматирано четене/писане**
- данните **най-често не са записани последователно като текст**

Затова използваме функциите:

- `istream& read(char* s, streamsize n);` – за четене на сурови байтове
- `ostream& write(const char* s, streamsize n);` – за запис на сурови байтове

Функциите са дефинирани в:

- `<istream>` – за `read()`
- `<ostream>` – за `write()`

Когато работим с `fstream`, имаме достъп **и до двете функции**, защото той комбинира входен и изходен поток.

~~~.cpp
// Четене
#include <ifstream>

int main(void){
    std::ifstream is("File.bin", std::ios::in | std::ios::binary);
    if(!is.is_open()) return -1;

    int number;
    is.read(reinterpret_cast<char*>(&number), sizeof(number));
    if(!is) return -1;

    // Прочетохме 4 байта от файла и ги асоцирахме към паметта на number, не знаем дали прочетохме нещо смислено?

    is.close();
    return 0;
}
~~~

---

~~~.cpp
// Писане
#include <ofstream>

int main(void){
    std::ofstream os("File.bin", std::ios::out | std::ios::binary );
    if(!os.is_open()) return -1;

    int number = 1;
    os.write(reinterpret_cast<const char*>(&number), sizeof(number));

    // Във файла има записани 4 байта за числото 1

    os.close();
    return -1;
}
~~~  
  
**Лека вметка**
   
Както виждате се налага да използваме **reinterpret_cast**, който преобразува между два на пръв поглед несъвместими типове. В действителност просто
се оказва на компилатора да интерпретира типа на адреса подаден в кръглите скоби, като типа подаден в ъгловите скоби. Не се извършва абсолютно никаква промяна на данните, като в практиката се използва при преобразуване на типове по указатели, тъй като запазва адреса на първоначалния указател, ето пример:
  
~~~.cpp
    int* a = new int(); // 0xA
    void* b = static_cast<void*>(a); // този е undefined според стандарта
    int* c = static_cast<int*>(b); // този пак ще е 0xA
~~~  
  
### Четене на блокове от данни 

**Масиви**   
  
Понякога ще ни се случва да искаме да запишем някакъв масив или последователност от елементи във файл. За това си задаваме въпроса - веднъж щом ги запиша, как 
мога да си прочта обратно данните. Отговорът е простичък, ако ще записваме масив, просто ще запишем размерът му преди самият масив, за да знаем колко точно да 
прочетем.

~~~.cpp
int main(void){
    int arr[] = {1,2,3};
    std::fstream stream("File.data", std::ios::in | std::ios::out | std::ios::binary);
    if(!stream.is_open()) return -1;

    int to_write = 3;
    stream.write(reinterpret_cast<const char*>(&to_write), sizeof(int));
    stream.write(reinterpret_cast<const char*>(arr), to_write * sizeof(int));

    stream.seekg(0, std::ios::beg); // двата указателя вървят заедно

    int size;
    stream.read(reinterpret_cast<char*>(&size), sizeof(int));

    int out[size]; // използвам Variable length стеков масив само за пример
    stream.read(reinterpret_cast<char*>(out), size * sizeof(int));

    stream.close();
    return 0;
}
~~~

**Структури**  
  
След като можем да прочтем масив, вече можем да се замисилм дали можем да запишем/прочетем и структура от двоичен файл? Отговорът е, да можем и то много лесно обаче ако структурата е **plain data** или казано по-просто, няма данни алокирани на heap-а, тъй като при запис във файл ще запишем адресът на указателя към динамичната структура, а тези указатели са само за текущата итерация на нашата програма, ако спрем програмата и опитаме да прочетем данните ще се натъкнем на **segmentation fault**, тъй като този адрес вече няма да пази нашите данни. За това как се справяме с този случай ще видите по-надолу!

~~~.cpp
#include <fstream>
#include <assert.h>
#include <string.h>

// това е plain data структура, подлежи на директен запис в двоичен файл
struct student{
    char name[32]{};
    int age{};
    int year{};
};

// тази структура не можем да запишем директно, имаме динамични данни...
struct cool_student{
    char* name{}; // --> указател
    int cool_data{};
};

std::streampos write_cool_data(cool_student student, std::fstream& stream){
    std::streampos pos = stream.tellp();

    int name_size = strlen(student.name);
    stream.write(reinterpret_cast<const char*>(&name_size), sizeof(name_size));
    stream.write(student.name, name_size);
    stream.write(reinterpret_cast<const char*>(&student.cool_data), sizeof(int));

    return pos;
}

bool read_cool_data(cool_student& student, std::fstream& stream, std::streampos pos){
    assert(!student.name);
    stream.seekg(pos, std::ios::beg);

    int size;
    stream.read(reinterpret_cast<char*>(&size), sizeof(int));

    student.name = new(std::nothrow) char[size + 1];
    if(!student.name) return false;

    stream.read(student.name, size);
    if(!stream) return false;

    stream.read(reinterpret_cast<char*>(&student.cool_data), sizeof(int));    
    if(!stream) return false;

    return true;
}

int main(){
    std::fstream stream("Data.bin", std::ios::in | std::ios::out | std::ios::binary | std::ios::trunc);
    if(!stream.is_open()) return -1;

    student pesho = {"Pesho", 10, 2};
    stream.write(reinterpret_cast<const char*>(&pesho), sizeof(pesho));
    // записахе | 32 байта низ | 4 байта инт | 4 байта int | + padding (0 в слуачая)

    student to_extract; stream.seekg(0, std::ios::beg);
    stream.read(reinterpret_cast<char*>(&to_extract), sizeof(student));

    // Как записваме структури от втори тип? Трябва да сериализираме на парчета...
    cool_student cool_pesho;
    cool_pesho.name = strcpy(new char[32], "Pesho"); 
    cool_pesho.cool_data = 42;

    std::streampos pos = write_cool_data(cool_pesho, stream);

    cool_student cool_pesho_extract; 
    read_cool_data(cool_pesho_extract, stream, pos);

    delete[] cool_pesho.name;
    delete[] cool_pesho_extract.name;
    return 0;
}
~~~
