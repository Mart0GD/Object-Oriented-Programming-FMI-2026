## Какво е полиморфизъм?

В същината на **обектно ориентираното програмиране** (ООП) е един принцип наречен Полиморфизъм (от гр. многообразен). Често ще ни се налага да искаме да обедим много 
различни по тип данни чрез обща за тях логика. Като различните езици за програмиране ни предоставят разнородни по мощност и удобство инстументи за точно тази работа.

**Function pointers**   
  
Нека дадем пример с много познатия за нас вече език **C**. Тук имаме набор от инструменти за постигане на полиморфно поведение - нека вземе указателите към функции. Ето 
едно приятно примерче, което илюстрира как може два различни обекта могат да са от еднакъв тип, но да имат различно поведение, в зависимост от имплементацията, 
с която ние сме ги снабдили:

~~~.cpp
#include <iostream>

struct machine{
    void (*start)(void);
};

void start_car() { std::cout << "Start car...\n"; }
void start_truck() { std::cout << "Start truck...\n"; }

int main(){
    machine car;
    car.start = start_car;
    
    machine truck;
    truck.start = start_truck;

    car.start();    // Prints start car...
    truck.start();  // Prints start truck...
    return 0;
}
~~~  
  
**Macros**  
     
Друг инструмент, с който разполагаме са макросите. Те ни дават възможност да имплементираме поведение, което зависи от множество фактори - други дефинирани макроси,
тип на подаден параметър, логика...   
Макросите са един изключително силен инструмент, който ни дава възможност да регулираме поведението на нашата програма спрямо конкретна логика. Ето един нагледен пример 
за логика, зависеща от операционната система, на която се намираме:  
  
~~~.cpp
#ifdef _WIN64
// special windows logic  
#elif __linux__
// special linux logic
#elif __unix__
// spcial unix based OS logic here
#endif
~~~

---

Това бяха два много добри примера за **динамичен** и **статичен полиморфизъм**. Разликата между двете както може би можете да се досетите е в това, по кое време на 
изпълнението на нашата програма знаем с какво работим или какво ще вършим. При динамичния полиморфизъм типа/логиката за изпълнение ще разберем по време на изпълнение на 
нашата програм (runtime), а при статичния полиморфизъм типа/логиката е ясно още по време на компилация.    
  
Както видяхте и може би пробвахте на вашите машини, при примера с макросите още по време на комилация знаем, че дефинираният от компилатора (в моя случай gcc)
флаг за операционна система ще бъде вдингат и ще имаме достъп до логиката в конкретния блок. 

За сега няма да споменавам повече за динамичния полиморфизъм, тъй като с тази концепция ще се занимаваме усилено след няколко упражнения. Обектът на това упражнение ще бъде
статичния полиморфизъм и какви удобства ни предоставя C++ за реализирането му под формата на така наречените **шаблони или "templates" на английски**...

## Шаблони, що е то?

Шаблоните са един удобен, начин посредством който можем да реализираме статичен полиморфизъм в C++, те възникват поради нуждата от по-лесен начин за имплементиране 
на полиморфно поведение, което можем и да дебъгваме сравнително лесно, тъй като дебъг процесът с макроси е почти напълно невъзможен. Въпреки това шаблоните са стриктно 
подмножество на макросите, тъй като те са в пъти по-мощни като инструмент...

**Дефиниране на шаблон**
  
Когато искаме да дефинираме шаблон за някаква функция/структура ще трябва да следваме следния синтаксис:

~~~.cpp
// Обща структура на шаблон
template< template_param1, template_param2, ... >
~~~

**Template_param** са специални типове само за шаблони, те могат да бъдат - **типови**, **константни** или **шаблонни**. 

Много е важно, че когато пишем шаблон, ние пишем рецепта и дотам. Конкретна имплементация не съществува. В момента, в който дефинираме инстанция
с **валидни** стойности за всеки параметър, тогава вече се създава конкретна **неявна специализация(implicit initialization)** на обекта в translation unit-а 
на файла, където сме го написали. Това означава, че ако имаме n еднакви шаблонни дефиниции на обект, но в различни файлове, то това генерира един и същи код
n пъти, но тъй като линкера е умен се изполва само **1**.




**Типове шаблонни параметри**
~~~.cpp
// Пишем всичко в .hpp или .inl файл

/* Using type template parameters
*  template< class/typename <name>, ...>            --> нормална дефиниция
*  template< class/typename <name> = <type> >       --> дефиниция със стойност по подразбиране
*  template< class/typename... <name> >             --> дефиниция с незнаен брой типове
*/

template<typename T>
inline T add(const T& a, const T& b) { return a + b;}

template<typename A = int, typename B = int>
inline void add(A& a, const B& b) { a += b;}

template<typename T = int> 
struct point { T x, t; };

template<typename T1, typename T2>
struct tuple { T1 val1; T2 val2; };


/* Using constant template parameters 
*  template< <type> <name>, ...>                --> нормална дефиниция
*  template< <type> <name> = <value> >          --> стойност по подразбиране 
*  template< <type>... <name> >                 --> дефиниция с незнаен брой стойности 
*  името никъде не е задължително
*/

template<const int amount = 2>
inline void adder(int& value) { a += amount; }

template<typename T, bool (*condition)(T)>
inline void check_array(const T& array, int len)
{
    for (i i = 0; i < len; i++)
        if(condition(array[i])) return true;
    return false;
}

template<typename T, T* arr>
inline bool create_array_of_10_elements() { arr = new T[10]; }

template<typename T, int size>
struct array {
    // Статичен масив
    T* arr = new T[size];    
};

// ** Creating a variable template **

template<typename T>
inline const T const_template = T(3.14f);
~~~

Пред всеки параметър стои думичката **typename/class**, може да използваме коя да е от двете. 
  
Когато пишем каквите и да е шаблони ще пишем в **.hpp** файл или ако искате да отделите логиката си в **.inl** или някой от поддържаните разширения за шаблони, 
като **задължително** трябва да добавите логиката обратно в края на файла. 
  
Също така пишем inline пред всяка дефиниция на шаблонна функция, за да не нарушим **one definition** правилото, което гласи, че всяка променлива/функция/обект трябва да 
имат една единствена дефиниция!

За темплейтните параметри пример не съм посочил, защото нямат адекватна полза за целите на курса ни, ако искате можете да прочетете сами за тях.
