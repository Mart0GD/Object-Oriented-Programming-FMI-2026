## Какво е полиморфизъм?

В същината на **обектно ориентираното програмиране** (ООП) е един принцип наречен Полиморфизъм (от гр. многообразен). Често ще ни се налага да искаме да обедим много различни по тип данни чрез обща за тях логика. Като различните езици за програмиране ни предоставят разнородни по мощност и удобство инстументи за точно тази работа.

**Function pointers**   
  
Нека дадем пример с много познатия за нас вече език **C**. Тук имаме набор от инструменти за постигане на полиморфно поведение - нека вземе указателите към функции. Ето едно приятно примерче, което илюстрира как може два различни обекта могат да са от еднакъв тип, но да имат различно поведение, в зависимост от имплементацията, с която ние сме ги снабдили:

~~~.cpp
#include <iostream>

struct machine{
    void (*start)(void);
};

void start_car() { std::cout << "Start car...\n"; }
void start_truck() { std::cout << "Start truck...\n"; }

int main(){
    machine car;
    car.start = start_car;
    
    machine truck;
    truck.start = start_truck;

    car.start();    // Prints start car...
    truck.start();  // Prints start truck...
    return 0;
}
~~~  
  
**Macros**  
     
Друг инструмент, с който разполагаме са макросите. Те ни дават възможност да имплементираме поведение, което зависи от множество фактори - други дефинирани макроси, тип на подаден параметър, логика...   
Макросите са един изключително силен инструмент, който ни дава възможност да регулираме поведението на нашата програма спрямо конкретна логика. Ето един нагледен пример за логика, зависеща от операционната система, на която се намираме:  
  
~~~.cpp
#ifdef _WIN64
// special windows logic  
#elif __linux__
// special linux logic
#elif __unix__
// spcial unix based OS logic here
#endif
~~~

---

Това бяха два много добри примера за **динамичен** и **статичен полиморфизъм**. Разликата между двете както може би можете да се досетите е в това, по кое време на изпълнението на нашата програма знаем с какво работим или какво ще вършим. При динамичния полиморфизъм типа/логиката за изпълнение ще разберем по време на изпълнение на нашата програм (runtime), а при статичния полиморфизъм типа/логиката е ясно още по време на компилация.    
  
Както видяхте и може би пробвахте на вашите машини, при примера с макросите още по време на комилация знаем, че дефинираният от компилатора (в моя случай gcc)
флаг за операционна система ще бъде вдингат и ще имаме достъп до логиката в конкретния блок. 

За сега няма да споменавам повече за динамичния полиморфизъм, тъй като с тази концепция ще се занимаваме усилено след няколко упражнения. Обектът на това упражнение ще бъде
статичния полиморфизъм и какви удобства ни предоставя C++ за реализирането му под формата на така наречените **шаблони или "templates" на английски**...

## Шаблони, що е то?

Шаблоните са един удобен, начин посредством който можем да реализираме статичен полиморфизъм в C++, те възникват поради нуждата от по-лесен начин за имплементиране на полиморфно поведение, което можем и да дебъгваме сравнително лесно, тъй като дебъг процесът с макроси е почти напълно невъзможен. Въпреки това шаблоните са стриктно подмножество на макросите, тъй като те са в пъти по-мощни като инструмент...

**Дефиниране на шаблон**
  
Когато искаме да дефинираме шаблон за някаква функция/структура ще трябва да следваме следния синтаксис:

~~~.cpp
// Обща структура на шаблон
template< template_param1, template_param2, ... >
~~~

**Template_param** са специални типове само за шаблони, те могат да бъдат - **типови**, **константни** или **шаблонни**. 

Много е важно, че когато пишем шаблон, ние пишем рецепта и дотам. Конкретна дефиниция не съществува. В момента, в който дефинираме инстанция
с **валидни** стойности за всеки параметър, тогава вече се създава конкретна **неявна специализация(implicit initialization)** на обекта в translation unit-а 
на файла, където сме го написали. Това означава, че ако имаме n еднакви шаблонни дефиниции на обект, но в различни файлове, то това генерира един и същи код
n пъти, но тъй като линкера е умен се изполва само **1**.

**Типове шаблонни параметри**
~~~.cpp
// Пишем всичко в .hpp или .inl файл

/* Using type template parameters
*  template< class/typename <name>, ...>            --> нормална дефиниция
*  template< class/typename <name> = <type> >       --> дефиниция със стойност по подразбиране
*  template< class/typename... <name> >             --> дефиниция с незнаен брой типове
*/

template<typename T>
inline T add(const T& a, const T& b) { return a + b;}

template<typename A = int, typename B = int>
inline void add(A& a, const B& b) { a += b;}

template<typename T = int> 
struct point { T x, t; };

template<typename T1, typename T2>
struct tuple { T1 val1; T2 val2; };


/* Using constant template parameters 
*  template< <type> <name>, ...>                --> нормална дефиниция
*  template< <type> <name> = <value> >          --> стойност по подразбиране 
*  template< <type>... <name> >                 --> дефиниция с незнаен брой стойности 
*  името никъде не е задължително
*/

template<const int amount = 2>
inline void adder(int& value) { a += amount; }

template<typename T, bool (*condition)(T)>
inline void check_array(const T& array, int len)
{
    for (i i = 0; i < len; i++)
        if(condition(array[i])) return true;
    return false;
}

template<typename T, T* arr>
inline bool create_array_of_10_elements() { arr = new T[10]; }

template<typename T, int size>
struct array {
    // Статичен масив
    T* arr = new T[size];    
};

// ** Creating a variable template **

template<typename T>
inline const T const_template = T(3.14f);
~~~

Пред всеки параметър стои думичката **typename/class**, може да използваме коя да е от двете. 

Също така пишем inline пред всяка дефиниция на шаблонна функция, за да не нарушим **one definition** правилото, което гласи, че всяка променлива/функция/обект трябва да имат една единствена дефиниция!

За темплейтните параметри пример не съм посочил, защото нямат адекватна полза за целите на курса ни, ако искате можете да прочетете сами за тях.

Сигурно сте забелязали, че когато пишем функции, имплементациите са дефинирани директно в .hpp или .inl файла. В ето това примерче нагледно можете да видите защо трябва да бъде така и защо не бива да пишем дефиницията на какъвто и да е шаблонен обект ако не искаме той да е достъпен само в translation unit-а на своя собствен .cpp файл 

~~~.cpp
//funcs.hpp
template<typename T>
void foo(); // само декларация

//funcs.cpp
template<typename T>
void foo() {...} // тук е дефиницията на функцията

//main.cpp
#include<funcs.hpp>

int main(void){
    foo<int>();
    /* Какво става тук?
    *  Имаме декларацията на foo няма да ни се скара директно
    *  Искаме функцията, като генерираме специализация за int, отиваме в header-а и търсим къде да вземем рецептата
    *  Обаче в funcs.cpp foo() проверява дали някой иска имплементация на нещо, никой в тукашният translation unit не я иска
    *  Не се генерира машинен код -> header-а няма какво да даде на foo в main.cpp и следва linking грешка!!!! 
    */
}
~~~

**Видове специализации на шаблони**

Както вече сте забелязали вече няколко пъти използвах думата специализация. Специализацията представлява инициализация на шаблонна дефиниция с конкретни шаблонни типове. **Оказваме на рецептата, че трябва да съществува функция с конкретни типове!**
  
Когато говорим за шаблони разполагаме с три вида специализации - **имплицитни**, **експлицитни** и **частични**. Фокусът ни ще падне върху първите две...

**Имплицитна специализация**  
  
Това е всяка една специализация, която се създава автоматично, когато инстанцираме обект или извикаме функция с дадени шаблонни параметри. Когато компилаторът види нещо от сорта на `my_struct<type>`, той проверява дали съществува експлицитна специализация и ако не намери такава, създава нова дефиниция на структурата с конкретния тип като шаблонен параметър.

~~~.cpp
// декларация и дефиниция на шаблонен тип
template<typename T>
struct my_struct { T data; };

int main(){
    /* създава се кода
    *  struct my_struct { int data; }
    *  и се поставя в тукашния translation unit
    */
    my_struct<int> s;
}
~~~

**Експлицитна специализация**

Това е специализация, която ние сами пишем с цел имплементиране на конкретна логика за определени типове. Може да искаме конкретна структура да оптимизира съхранението на данните си спрямо типа им, може и да искаме конкретна функция да работи различно спрямо специфични типове. Тогава създаваме експлицитни специализации. 
Имаме няколко ограничения при създаването на експлицитни специализации:
  
**Не можем да създадем ЕС за тип, за който вече има генерирана имплицитна такава.**
    
~~~.cpp
//.hpp
template<typename T>
inline void sort(const T* arr, int size) { ... }

//.cpp
void foo(int* arr, int size) {
    sort(arr, size);
}

template<>
sort<int>(const int* arr, int size); // невалидно, вече има специализация за int
~~~

**Не можем да създадем ЕС ако декларацията на шаблонния тип не се намира в същия контекст.**

~~~.cpp
namespace array{
    template<typename T>
    struct arr { T* data; int elems; }
};

template<>
arr<int> a; // не е в същия контекст      
~~~

**Можем да пропуснем <> указващи типа при експлицитна специализация на функции, ако може да се подразбере от типа на параметрите**

~~~.cpp
template<typename T>
void foo(T a, T b);

template<>
void foo(int a, int b); // --> типовете се подразбират
~~~

**Частична специализация**

Частичните специализации са нови шаблонни типове, надграждащи над други с по-конкретни специализации на някои от шаблонните параметри, но все още не са конкретни дефиниции, тъй като някои от типовете още не са зададени. Важното при тях е, че трябва да се запишат след шаблонната дефиниция и трябва да са в един и същи контекст...

~~~.cpp
template<class T1, class T2, int I>
class A {};             // primary template
 
template<class T, int I>
class A<T, T*, I> {};   // #1: partial specialization where T2 is a pointer to T1
 
template<class T, class T2, int I>
class A<T*, T2, I> {};  // #2: partial specialization where T1 is a pointer
 
template<class T>
class A<int, T*, 5> {}; // #3: partial specialization where
                        //     T1 is int, I is 5, and T2 is a pointer
 
template<class X, class T, int I>
class A<X, T*, I> {};   // #4: partial specialization where T2 is a pointer
~~~
